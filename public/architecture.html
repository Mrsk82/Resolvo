🔐 Security Architecture
Every security layer protecting Resolvo and its brands
Authentication Layers
Password Security
→bcrypt cost factor 10 — ~100ms hash time
→Legacy plain-text auto-migrated on first login
→Min 6 chars enforced at API level
→Reset tokens stored in owner.json (persistent)
→Reset tokens expire in 1 hour
→Token deleted after single use
→mustChangePassword flag for first-time users
Session Security
→UUID v4 session tokens (128-bit entropy)
→8 hour TTL (30 days with remember-me)
→Stored in memory Map — server restart = logout
→Owner can force-kill any session
→Brand slug baked into session (cannot change)
→Feature flags resolved at login time
→isOwner flag — separate owner vs brand sessions
Rate Limiting
→5 failed login attempts per IP per 15 minutes
→Stored in memory loginAttempts{}
→IP from req.ip (trust proxy enabled for ngrok)
→Returns 429 with retry-after message
→Auto-clears after 15 minute window
Two-Factor Auth (2FA)
→TOTP (Time-based One-Time Password)
→Compatible with Google Authenticator, Authy
→QR code generated via QR server API
→30-second window (±1 token accepted)
→Secret stored as base32 in user object
→Disable requires current password verification
Data Isolation
Complete brand isolation: Every API call uses su.brandSlug from the session token to scope all database reads/writes. A brand user physically cannot read another brand's data — the slug is baked into their session at login and cannot be changed by the client.
LayerIsolation Mechanism
DatabaseSeparate SQLite file per brand: data/brands/:slug/db.sqlite
SessionbrandSlug in session token — set at login, immutable
API callsconst slug = su.brandSlug — all reads/writes use this
EmailsendBrandEmail(slug,...) uses brand's own SMTP
Feature flagsResolved per brand at login — different for each brand
IMAP pollersOne cron job per brand slug — complete separation
